# Source - Claude Code Monitoring

[[Claude Code]] exports telemetry data via [[OpenTelemetry]] (OTel), covering metrics, events/logs, and optionally distributed traces. Telemetry is opt-in and requires setting `CLAUDE_CODE_ENABLE_TELEMETRY=1`. Metrics are exported as time-series data; events/logs use the OTel logs/events protocol; traces use the traces protocol (beta). Administrators can push OTel configuration to all users via managed settings files, which override user-level settings and can be distributed through MDM solutions.

Eight metrics are exported: `claude_code.session.count` (sessions started), `claude_code.lines_of_code.count` (lines added/removed), `claude_code.pull_request.count` (PRs created), `claude_code.commit.count` (commits created), `claude_code.cost.usage` (USD cost per API request), `claude_code.token.usage` (input/output/cache tokens), `claude_code.code_edit_tool.decision` (accept/reject decisions for Edit/Write/NotebookEdit), and `claude_code.active_time.total` (active seconds, split by user keyboard vs. CLI processing). Four event types are exported: `claude_code.user_prompt`, `claude_code.tool_result`, `claude_code.api_request`, and `claude_code.api_error`. Events carry a `prompt.id` UUID correlating all events triggered by a single prompt. User prompt content and tool arguments are redacted by default and must be opted into with `OTEL_LOG_USER_PROMPTS=1` and `OTEL_LOG_TOOL_DETAILS=1`.

Distributed tracing (beta) is enabled by additionally setting `CLAUDE_CODE_ENHANCED_TELEMETRY_BETA=1` and `OTEL_TRACES_EXPORTER`. Traces link each user prompt to its downstream API requests and tool executions. When tracing is active, Bash and PowerShell subprocesses automatically inherit a `TRACEPARENT` environment variable containing the W3C trace context, enabling end-to-end distributed tracing through scripts Claude runs. Tool content in spans is redacted by default and requires `OTEL_LOG_TOOL_CONTENT=1`. Supported metrics exporters include `otlp`, `prometheus`, and `console`; supported backends include Prometheus, ClickHouse, Honeycomb, Datadog, Jaeger, Zipkin, Grafana Tempo, and Elasticsearch/Loki.

Cardinality can be managed by toggling `OTEL_METRICS_INCLUDE_SESSION_ID`, `OTEL_METRICS_INCLUDE_VERSION`, and `OTEL_METRICS_INCLUDE_ACCOUNT_UUID`. Multi-team organizations can add custom resource attributes via `OTEL_RESOURCE_ATTRIBUTES` for department- or cost-center-level segmentation. Dynamic authentication headers for enterprise OTel collectors can be generated by a helper script configured in `otelHeadersHelper` (settings.json), refreshed every 29 minutes by default.

## Key Topics

- `CLAUDE_CODE_ENABLE_TELEMETRY=1` required to activate telemetry (opt-in)
- Metrics exporters: `otlp`, `prometheus`, `console` (via `OTEL_METRICS_EXPORTER`)
- Events/logs exporter via `OTEL_LOGS_EXPORTER`; default export interval 5 seconds
- 8 exported metrics: sessions, lines of code, PRs, commits, cost (USD), tokens, tool decisions, active time
- 4 event types: user_prompt, tool_result, api_request, api_error
- `prompt.id` UUID for correlating all events from a single user prompt
- User prompt content redacted by default; opt-in via `OTEL_LOG_USER_PROMPTS=1`
- Tool arguments redacted by default; opt-in via `OTEL_LOG_TOOL_DETAILS=1`
- Beta distributed tracing via `CLAUDE_CODE_ENHANCED_TELEMETRY_BETA=1`
- W3C `TRACEPARENT` propagation to Bash/PowerShell subprocesses
- Tool content in spans opt-in via `OTEL_LOG_TOOL_CONTENT=1` (truncated at 60 KB)
- Supported backends: Prometheus, ClickHouse, Honeycomb, Datadog, Jaeger, Zipkin, Grafana Tempo
- Admin-managed telemetry config via managed settings files (MDM-distributable)
- `otelHeadersHelper` script for dynamic authentication header generation
- `OTEL_RESOURCE_ATTRIBUTES` for multi-team department/cost-center segmentation
- Cost metrics are approximations; official billing from provider dashboards
